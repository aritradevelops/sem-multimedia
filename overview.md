* ASCII (American Standard Code for Information Interchange) was designed with 7 bits primarily due to the technological constraints and priorities of the time it was developed (the 1960s).                  Reasons for 7-bit ASCII:     Sufficiency for English: 7 bits allowed for 128 unique combinations (\(2^{7}=128\)), which was sufficient to represent all the characters commonly used in American English, including uppercase and lowercase letters, digits, punctuation marks, and control characters (like tab, newline, etc.). Minimizing Data Transmission Costs: In an era of slower and more expensive data transmission, using 7 bits instead of 8 bits meant sending less data, thus reducing costs and improving transmission speed. Parity Bit for Error Checking: The 8th bit, if present in an 8-bit byte, could be used as a parity bit for basic error detection during data transmission, which was crucial given the less reliable communication equipment of the time. Backward Compatibility with Teletype Systems: ASCII aimed to replace older encoding schemes like the 5-bit Baudot code used in teletype machines, and a 7-bit standard offered a significant improvement while still being relatively efficient.¬†              While 8-bit bytes became common later, ASCII's 7-bit design was a pragmatic choice that addressed the specific needs and limitations of computing and communication technology in the mid-20th century. The 8th bit later found use in "extended ASCII" variants to include additional characters for other languages or symbols.

# üéö **BIT DEPTH ‚Äî DEEP BUT INTUITIVE EXPLANATION**

Bit depth = **how precisely each audio sample‚Äôs amplitude is stored**.

You already know sampling rate = *how often* we sample.
Bit depth = *how accurately* we store each sample.

---

# üß† **Think of bit depth like the number of shades in a grayscale image**

* 1-bit image ‚Üí only black or white (2 levels)
* 2-bit image ‚Üí 4 shades
* 8-bit image ‚Üí 256 shades
* 16-bit image ‚Üí 65,536 shades

More shades = smoother image
Less shades = banding

Similarly in sound:

More bit depth ‚Üí smoother wave
Low bit depth ‚Üí noisy, harsh sound

---

# ‚≠ê WHY BIT DEPTH MATTERS?

It affects **three** things:

1. **Dynamic Range**
2. **Noise Floor**
3. **Accuracy of the waveform**

Let‚Äôs break these down.

---

# 1Ô∏è‚É£ **Dynamic Range**

Dynamic range = difference between the **quietest** and **loudest** sound that can be recorded.

Formula you‚Äôll see in textbooks:

### **Dynamic Range ‚âà 6.02 √ó bit depth (in dB)**

(You don‚Äôt have to memorize, just understand.)

Examples:

| Bit Depth | Levels       | Dynamic Range | What It Means                       |
| --------- | ------------ | ------------- | ----------------------------------- |
| 8-bit     | 256          | ~48 dB        | mediocre quality, telephone quality |
| 16-bit    | 65536        | ~96 dB        | CD-quality audio                    |
| 24-bit    | 16.7 million | ~144 dB       | studio-level recording              |

### Meaning:

* 16-bit gives enough range to capture quiet whispers and loud music.
* 8-bit cannot ‚Äî quiet sounds vanish into noise.

---

# 2Ô∏è‚É£ **Quantization Noise (Key concept!)**

Remember:
When we sample a waveform, the amplitude is a continuous analog value.
But computers can only store **discrete levels**.

Example:
If the real amplitude is 2.76 but the nearest digital value is 3 ‚Üí **rounded up**.
or
If it‚Äôs 2.49 ‚Üí **rounded down** to 2.

This rounding error = **quantization noise**.

### More bit depth ‚Üí more levels ‚Üí smaller rounding error ‚Üí cleaner audio.

---

# 3Ô∏è‚É£ **Accuracy of Waveform (resolution)**

Higher bit depth gives:
‚úî better representation of quiet sounds
‚úî smoother fades
‚úî more precise reverb/echo
‚úî no ‚Äúgraininess‚Äù or ‚Äúbuzzing‚Äù in quiet passages

This is especially important in:

* classical music
* film scoring
* ambient sound design
* podcast editing (soft voice parts)

---

# üéß **REAL-WORLD EXAMPLES YOU CAN HEAR**

### **8-bit audio** (old video games)

* Grainy
* Harsh
* Noticeable buzzing
* Quiet parts sound like ‚Äústatic + tone‚Äù

### **16-bit audio (CD quality)**

* Smooth and clear
* No audible quantization noise
* Perfect for music playback

### **24-bit audio (studio)**

* Better for editing, because:

  * You can record very quiet details
  * Apply effects without degrading quality
  * Higher headroom (won‚Äôt clip easily)

### Important point:

When exporting to MP3/AAC, most music ultimately ends up as **16-bit PCM ‚Üí compressed**.

---

# üß© **PUTTING IT ALL TOGETHER**

Bit depth determines:

### ‚úî Volume detail

Quietest ‚Üí loudest differences

### ‚úî Precision

How accurately waveforms are stored

### ‚úî Noise

Lower bit depth ‚Üí more quantization noise

24-bit for production, 16-bit for distribution

# üì¶ **FILE SIZE and BIT DEPTH**

Formula (important for exam numericals):

```
File Size = Sampling Rate √ó Bit Depth √ó Channels √ó Duration
```

Example:
44.1 kHz, 16-bit, stereo, 10 seconds:

= 44,100 √ó 16 √ó 2 √ó 10
= 14,112,000 bits
‚âà 1.76 MB

You don‚Äôt need to memorize, but know how bit depth affects file size.

---

# üóÇ **NUMBER OF LEVELS (Exam Question!)**

Formula:

```
Levels = 2^(bit depth)
```

Examples:

* 8-bit ‚Üí 256 levels
* 16-bit ‚Üí 65,536 levels
* 24-bit ‚Üí 16,777,216 levels

This shows why 24-bit audio is so smooth ‚Äî millions of possible amplitude values.

---

# üß™ **An intuitive analogy**

Imagine drawing a circle pixel by pixel:

* 8 pixels = jagged shape
* 64 pixels = smoother circle
* 1000+ pixels = looks perfect

Bit depth is exactly like increasing the number of pixels vertically in the waveform.

# üóúÔ∏è **1. IMAGE COMPRESSION ‚Äì The Intuitive Deep Dive**

## PART A ‚Äî JPEG (Lossy Compression)

JPEG is optimized for **real-world photos**, not text/logos.

### üåà Key Idea:

JPEG removes details that **your eyes won‚Äôt notice**.

Let‚Äôs go step by step.

---

## üß© **Step 1: Split the image into 8√ó8 blocks**

A photo becomes puzzle pieces:

```
+--------+--------+
| 8√ó8 px | 8√ó8 px |
+--------+--------+
| 8√ó8 px | 8√ó8 px |
+--------+--------+
```

Why 8√ó8?
Because this size balances:

* accuracy
* speed
* compression efficiency

---

## üß† **Step 2: Convert RGB ‚Üí YCbCr**

JPEG doesn‚Äôt store RGB.

Instead it stores:

* **Y** = brightness
* **Cb** = blueness
* **Cr** = redness

Why?

Because humans notice changes in **brightness** more than **color**, so JPEG can heavily compress color channels.

---

## üß± **Step 3: Apply DCT (Discrete Cosine Transform)**

This is the heart of JPEG.

Think of DCT as breaking an image block into **patterns of waves**:

Example:

* Low-frequency waves ‚Üí smooth areas (sky)
* High-frequency waves ‚Üí details (edges, hair)

ASCII idea:

```
Low freq: ~~~
High freq: ^v^v^v^v
```

DCT converts the 8√ó8 block from pixels ‚Üí frequency components.

---

## ‚úÇÔ∏è **Step 4: Quantization (where data is lost)**

High-frequency values get divided by large numbers ‚Üí becoming **zero**.

Example:

```
Before quantization:  12  5  0  -3  1
Divide by Q matrix ‚Üí   3  1  0   0  0
```

Your eyes don‚Äôt notice high-frequency removal ‚Üí compression achieved!

---

## üîê **Step 5: Huffman coding**

Finally, JPEG uses entropy compression to squeeze the numbers further.

This step is lossless.

---

## üéØ Result:

* Great for *photos*
* Bad for *text, logos, UI* ‚Üí causes blocky artifacts

---

# PART B ‚Äî PNG (Lossless Compression)

PNG aims to **never lose data**.

### Why PNG is huge for photos?

Because photos don‚Äôt have repeating patterns ‚Üí compression is weak.

---

## PNG Compression Steps

### üñåÔ∏è **Step 1: PNG applies ‚Äúfilters‚Äù row-by-row**

A filter predicts the next pixel, stores the **difference** instead.

Example:

```
Pixel:     100 102 105 107 108
Stored:      0   2   3   2   1   (differences)
```

Differences compress better than actual pixel values.

---

### üì¶ **Step 2: DEFLATE (LZ77 + Huffman)**

PNG then applies:

* **LZ77** ‚Üí replaces repeated patterns
* **Huffman coding** ‚Üí compresses frequently used values

Since no data is lost, the image is reconstructed **bit-for-bit**.

---

## When to use PNG?

‚úîÔ∏è Icons
‚úîÔ∏è Logos
‚úîÔ∏è UI assets
‚úîÔ∏è Transparency (RGBA)
‚úñÔ∏è Not for natural photos (file huge)

# üîó **1. What is Synchronization?**

In multimedia, **synchronization** means keeping different media streams **aligned in time**.

Example:

* Your mouth moves ‚Üí sound must match
* Background music must start at the right moment
* Slides must change when the presenter speaks

If they misalign, the experience breaks.

---

# üïí **2. Temporal Relationships**

These describe how different media streams relate in time.

### **Simultaneous**

Two streams occur at the same time.
Example: Video + matching audio.

### **Sequential**

One stream follows another.
Example: Intro music ‚Üí then the lecture video.

### **Conditional**

Media appears based on a user action.
Example: Clicking a button plays a sound.

These relationships define the *timing structure* of multimedia presentations.

---

# üéØ **3. Synchronization Types**

## **(A) Intra-stream synchronization**

Keeping timing correct *within the same stream*.

Example:

* Ensuring video frames play at correct fps (30fps/60fps)
* No duplicate or skipped frames

This is handled by the **decoder** and **media player clock**.

---

## **(B) Inter-stream synchronization**

Keeping timing correct *between streams*.

Examples:

* Lip-sync (audio + video)
* Slides and narration
* Sound effects matching animation

The system must ensure:

> The event in stream A occurs at the same time as stream B.

---

## **(C) Inter-object synchronization**

Multiple multimedia objects coordinated together.

Example:

* A video plays ‚Üí then after 5 seconds ‚Üí text appears ‚Üí then animation starts
  Used in:
* Interactive media
* Presentations

---

# üéØ **4. Synchronization Accuracy**

This is how precisely streams need to be aligned.

Examples:

| Application          | Required Accuracy   |
| -------------------- | ------------------- |
| Lip-sync             | < 80 ms             |
| Music videos         | < 20 ms             |
| Movies               | < 80‚Äì100 ms         |
| Voice-over on slides | Up to 200 ms OK     |
| Video conferencing   | 100‚Äì150 ms expected |

If delay exceeds this, users notice mismatch.

---

# ‚öôÔ∏è **5. Factors Affecting Synchronization**

1. **Network Jitter**
   Variable delays cause audio/video drift.

2. **Packet Loss**
   Missing packets cause gaps ‚Üí desynchronization.

3. **CPU Load**
   Slow decoding = video lags behind audio.

4. **Different Clocks**
   Audio and video devices running at slightly different speeds.

5. **Compression & Buffering**

   * Audio is faster to decode than video
   * Video buffering delays increase mismatch

---

# üß∞ **6. How systems maintain synchronization**

### **A. Time Stamps**

Every audio/video packet has a timestamp.
The player uses it to line up streams correctly.

Standards:

* RTP timestamps (video calling)
* MPEG PTS/DTS timestamps

### **B. Master Clock**

One stream is treated as "reference" (usually audio).
Other streams adjust to match it.

### **C. Buffers**

Small delays added to smooth jitter.

### **D. Frame Dropping / Duplication**

If video falls behind ‚Üí drop frames
If audio leads ‚Üí add silence
Goal: preserve sync.

---

# üß™ **7. Quality of Service (QoS)** (important for exam)

QoS ensures multimedia plays smoothly with correct timing.

Includes:

* guaranteed bandwidth
* minimal jitter
* limited delay
* good throughput

Used in:

* video conferencing
* VoIP
* streaming media

It includes intra-stream, inter-stream, and inter-object synchronization, affected by network jitter, buffering, processing delays, and controlled using timestamps, master clocks, and QoS mechanisms.**

# **4.1 Image Representation**

Image representation = how an image is stored inside a computer.

Two common ways:

### **1. Pixel-based representation (Raster images)**

* Image is a matrix of pixels.
* Each pixel has color values (RGB, CMYK, etc).
* Used in photographs, screenshots.

### **2. Feature-based representation**

Instead of storing every pixel, extract meaningful *features* such as:

* **Edges**
* **Corners**
* **Color histograms**
* **Textures**

Used for **searching** and **indexing** images.

---

# **4.2 Image Segmentation**

Segmentation = dividing an image into meaningful regions.

### Purpose

* Identify objects in an image
* Used in retrieval, medical imaging, surveillance

### Common Techniques

1. **Thresholding**

   * Separate foreground & background using intensity value.
   * Simple but effective for high-contrast images.

2. **Edge-based segmentation**

   * Use edge detection (Sobel, Canny).
   * Good for object boundaries.

3. **Region-based segmentation**

   * Region growing
   * Region merging
   * Split & merge (quadtree-based)

4. **Clustering-based segmentation**

   * k-means
   * Groups pixels with similar color/texture

---

# **4.3 Similarity-Based Retrieval**

When you search images by saying ‚Äúfind similar images,‚Äù the system needs similarity measures.

### **Ways to determine similarity**

#### **1. Color-based retrieval**

* Use color histograms
* Example: find all images with dominant blue color

#### **2. Shape-based retrieval**

* Uses object boundaries / contour descriptors
* Example: find all round-shaped objects

#### **3. Texture-based retrieval**

* Uses patterns like rough, smooth, grainy
* Methods:

  * Gabor filters
  * Co-occurrence matrices

These form the foundation of **CBIR (Content-Based Image Retrieval)**.

---

# **4.4 Indexing Structures for Multimedia Databases**

Because multimedia data is large and multi-dimensional, special indexing structures are used:

---

### **1. KD-Trees (k-dimensional trees)**

* Used for low-dimensional data (2D, 3D, up to ~20).
* Works well for color spaces (RGB ‚Üí 3D).
* Binary tree that partitions data along axes.

**Example use:**
Find nearest color match in an RGB color space.

---

### **2. R-Trees**

* Used for indexing spatial objects (rectangles, regions).
* Works well for images stored with bounding boxes.

**Application:**

* GIS
* Object-based image search
* Video keyframe indexing

---

### **3. Quad Trees**

* Recursively divide image into 4 equal parts.
* Used in image segmentation, compression.

**Key idea:**

* If a region is uniform ‚Üí store as a single block
* If not ‚Üí split into 4 and repeat
